An Investigation of Graph Databases

The demands of databases since the rise of the internet has required high concurrency and efficient storage and query performance.

Neo4j is a leading implementation of a Graph database that has been adopted by many large organizations such as eBay, Walmart and governments.

What are the benefits of Neo4j being a graph database?

With RDB's increasing the depth of a query (the number of joins/connections) dramatically reduces performance. This is because every table accessed need to be 'indexed' (lookup). With a graph database, the performance remains constant regardless of the number of connections between entities. This makes graph databases suitable for highly connected data.

It is also easier to update the model of a graph database on the fly as the graph database does not require a schema. This can reduce the need for database schema migrations.

The graphs in a graph database are composed of nodes and are connected by relationships. Nodes represent entities. This is a benefit because the database implementation can closely resemble a logical or physical entity relationship diagram.

Graph databases have a 'connection first approach to data storage. The relationships and connections between entities are permanently stored in graphs. In comparison with an RDBS, the connections are calculated during the query. This storage of connections/references is what enables constant performance regardless of depth. Because all connections are persisted there is no need for foreign keys and a reduced need for post-processing of data after querying.

Neo4j is built using native graph storage whereas some graph databases still use RDBS under the hood. Native graph storage increases performance. Graph databases are great for data analysis that involves lots of connections, as such, it has been used widely for fraud detection and product recommendations.

Part of the success of Neo4j's adoption is because it conforms to the ACID standard. This means it is a stable platform for critical information.

ACID

  • Acid is a standard for databases that requires the database to guarantee transactions are processed reliably.
  • Atomicity - either all parts of a transaction succeed or no parts of it should. This prevents fragmented data from being stored.
  • Consistency - All data must be valid according to defined constraints
  • Isolation - Every transaction must occur in isolation. One transaction cannot see changes made by another until the other is completed
  • Durability - Once a transaction is complete it must be stored permanently and immediately
  • Confirmation with ACID helps to prevent data corruption. Acid compliance reduces the damage of software and hardware crashes and failed transactions.

RDBS normally fully comply with ACID principles, it is more common for NoSQL databases not to fully comply with ACID. Not complying with ACID can offer possibilities for performance gains.

Many NoSQL databases make use of eventual consistency for performance gains.

Eventual consistency

Eventual consistency is a guarantee made by many distributed systems. It guarantees that if data is repeated across many nodes then changing one piece of data will mean all the other replicas are updated to match it.

The use of Eventual consistency means that data will often be out of sync enabling the possibility of retrieving out of date information.

Neo4j does not use eventual consistency because the nature of graph storage requires strong consistency. Because of this requirement up until recently, Neo4j has been a 'single image system'. This means it runs from a single server. Running on a single machine helped ensure strong consistency as it eliminated the time for networking calls.

As of early 2020 support has been added to Neo4j for running across distributed systems. In this implementation, each system will store a subgraph of the full graph. Neo4j are able to maintain strong consistency even when running on distributed systems by using collision avoidance so that only a small fraction of transactions need to be coordinated.

Reasons not to use graph databases

  • If the data model you are working with is likely to remain very static and tabular then you will not receive the full benefits of a graph databases flexible approach and an RDBS can better ensure data integrity.
  • Graph databases are efficient when starting from a specific node/entity but are slow at traversing large sets. As such if you will be using queries that require traversing large sets graph databases are not optimal.
  • Graph databases are bad at storing large blobs of data as attributes. Large blob sizes slow down traversal. Perhaps a key-value pair would be more suitable in this instance
